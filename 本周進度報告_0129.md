## 第一部分：TracIn 診斷完整分析

### 1.1 實驗目標

使用 TracIn (Training Data Attribution) 方法追溯 validation set 上的 token collapse 失敗樣本，分析哪些 training samples 對這些失敗貢獻最大，找出 root causes 並提出修正方向。

### 1.2 技術改進：5-Checkpoint 分析

**動機**：提高 TracIn 分析的穩健性和可信度

**方法**：
- **擴充 checkpoint 數量**：從 2 個增加到 5 個
- **選擇策略**（依 TracIn 論文建議）：
  - epoch010：訓練初期（學習 general patterns）
  - epoch050：warmup 結束（loss 下降最快期）
  - epoch150：curriculum 過渡期中段
  - epoch250：warmdown 開始
  - epoch300：訓練末期（final model）

**成果**：
- 產出 [tracin_scores_5ckpt.csv](exp_0125/tracin_token_collapse_589e6d/tracin_scores_5ckpt.csv) (10,001 rows)
- 產出 [profiles_5ckpt/](exp_0125/tracin_token_collapse_589e6d/profiles_5ckpt/) 詳細分析
- 更新完整分析報告 [CONCLUSION.md](exp_0125/tracin_token_collapse_589e6d/CONCLUSION.md)

### 1.3 核心診斷結果

#### Proponents vs Opponents Profile

TracIn 識別出對 validation collapse 有正向/負向影響的訓練樣本特徵：

| 指標 | Proponents (Top-100) | Opponents (Top-100) | 全體 Train | Cohen's d |
|------|----------------------|---------------------|------------|-----------|
| **SNR (dB)** | **-2.24** | -0.68 | -1.88 | -0.107 (小效應) |
| **Noise: papercup** | **57%** | 9% | 33% | 1.7x 過度代表 |
| **Noise: box** | 29% | **58%** | 33% | - |
| **Noise: plastic** | 14% | 33% | 33% | - |

**關鍵發現**：
1. ✅ **Proponents 偏向低 SNR**：比全體 train 低 0.36 dB
2. ✅ **Proponents 以 papercup 材質為主**：57% vs 33% (1.7倍)
3. ✅ **Opponents 偏向 box 材質**：58% vs 33%，且 SNR 較高

**視覺化證據**（5-checkpoint TracIn）：

![TracIn Influence vs SNR](exp_0125/tracin_token_collapse_589e6d/profiles_5ckpt/plots/influence_vs_snr.png)

*圖 1：TracIn influence scores vs SNR。可見高 influence 樣本（proponents）集中在低 SNR 區域（-8 ~ -4 dB）*

#### 穩健性驗證

**L_anchor 版本交叉驗證**（2-checkpoint 參考）：
- Proponents SNR: **-2.67 dB** (Cohen's d = -0.292)
- Proponents papercup: **62%**
- **結論**：L_anchor 結果與 L_train 一致，支持診斷穩健性

**音質交叉檢查**：
- 對 bottom-PESQ 子集（音質最差 30 筆）做 TracIn
- 結果：Proponents 仍指向 papercup **59%**、SNR **-2.10 dB**
- **結論**：音質最差樣本的影響因素與整體一致

### 1.4 Counterfactual 驗證

**實驗設計**：移除 TracIn top-200 proponents，用剩餘 1,800 samples 做短跑訓練（800 steps）

**結果**：

| 指標 | Baseline | Counterfactual | 變化 |
|------|----------|----------------|------|
| Val Strict Acc | 0.91% | **0.82%** | ⬇ -0.09% |
| Val Entropy | 6.07 | **5.86** | ⬇ -0.21 |
| Val Top-10 Mass | 19.7% | **28.1%** | ⬆ +8.4% |

**重要結論**：
- ❌ 硬刪除高影響樣本會**惡化** collapse
- ✅ 這些樣本不是「壞資料」，而是「權重過高」
- ✅ 需要 **soft reweighting** 或 **噪音材質平衡**，而非硬刪

### 1.5 Root Causes 確認

**#1 H3：資料驅動 — 噪音材質強的 train 子集主導 LoRA 更新**

證據：
- Proponents SNR -2.24 dB (Cohen's d = -0.107)
- Proponents papercup 57% (vs 全體 33%)
- Opponents 偏向 box 58%, papercup 僅 9%

機制：LoRA 在低 SNR + papercup 材質的樣本上學得最多，這些樣本的 gradient 主導了參數更新方向。

**#2 H2：VQ margin 不穩定放大 collapse**

證據：
- VQ margin p50: Train 0.0169 vs Val **0.0109**
- VQ margin p90: Train 0.0619 vs Val **0.0311**

機制：Val 的 VQ margin 更小，量化決策更不穩定，容易塌縮到少數 token。

**#3 H1：Noise-dependent / joint encoding**

證據：
- 高 influence 偏向特定噪音（papercup 57%）
- 音質最差子集同樣指向 papercup 59%

機制：模型把噪音資訊編進 token（joint encoding），在 inference 時無法忽略噪音變化。

## 第二部分：Phase 1 驗證實驗（失敗分析）

### 2.1 實驗設計

基於 TracIn 診斷，測試兩種短期修復方案：

#### 實驗 1：TracIn-Weighted Soft Reweighting

**方法**：使用 TracIn influence scores 進行軟性重加權

```python
weight_i = 1 / (1 + α × TracIn_score_i)
```

- High influence proponents → 降低權重
- Low influence opponents → 提高權重
- **參數**：α = 0.5

**預期**：減少 proponents 過度影響，同時保留樣本多樣性

#### 實驗 2：Noise-Balanced Sampling

**方法**：強制每個 batch 的 noise material 平衡分佈

- box : papercup = 1 : 1 (原本 box 51.9%, papercup 48.1%)
- 解決 proponents 中 papercup 過度代表問題 (57% → 50%)

**預期**：平衡噪音分佈，降低 noise-dependent encoding 風險

#### 配置一致性

兩個實驗與 baseline (exp_k v6) **完全一致**，唯一差異為 sampler：

| 配置項 | exp_k v6 | 實驗 1 | 實驗 2 |
|--------|----------|--------|--------|
| LoRA (rank/alpha/dropout) | 256/512/0.2 | 256/512/0.2 | 256/512/0.2 |
| Intermediate layers | [3, 4, 6] | [3, 4, 6] | [3, 4, 6] |
| Optimizer | AdamW (1e-4/0.01) | AdamW (1e-4/0.01) | AdamW (1e-4/0.01) |
| Training steps | - | 1000 | 1000 |
| **Sampler** | **Random** | **TracInWeighted** | **NoiseBalanced** |

### 2.2 實驗結果：兩項均失敗

**最終結果對比**：

| 實驗 | Entropy | Top-10 Mass | Strict Acc | 判定 |
|------|---------|-------------|------------|------|
| Baseline (exp_k v6 @ epoch 300) | 6.07 | 19.7% | 0.91% | - |
| **實驗 1** (TracIn Weighted) | **5.63** ⬇ | **29.0%** ⬆ | **0.60%** ⬇ | ❌ 失敗 |
| **實驗 2** (Noise Balanced) | **5.56** ⬇ | **28.3%** ⬆ | **0.52%** ⬇ | ❌ 失敗 |

**所有指標均顯著惡化**：
- Entropy 下降 0.44-0.51 (應 ↑)
- Top-10 Mass 增加 8.6-9.3% (應 ↓)
- Strict Acc 下降 0.31-0.39% (應持平或 ↑)

### 2.3 震撼發現：未訓練狀態優於 Baseline

**兩實驗的 Step 0 (初始 LoRA) 指標**：

| Metric | Step 0 (未訓練) | Baseline (epoch 300) | 差異 |
|--------|----------------|---------------------|------|
| Entropy | **6.26** | 6.07 | **+0.19** ✅ 更好 |
| Top-10 Mass | **11.8%** | 19.7% | **-7.9%** ✅ 更好 |
| Strict Acc | **3.52%** | 0.91% | **+2.61%** ✅ 更好 |

**結論**：
- 隨機初始化的 LoRA 參數產生的編碼**優於**經過 300 epochs 訓練的模型
- **訓練過程本身導致 collapse**，而非數據分佈問題

### 2.4 訓練動態分析

**所有實驗的共同模式**：

```
Initial (Step 0):              Entropy ~6.26, Top-10 ~11.8%  ✅ 健康
    ↓ (200 steps)
Early Training:                Entropy ~5.5-5.7, Top-10 ~20-27%  ⚠️ 開始 collapse
    ↓ (800 steps)
Late Training:                 Entropy ~5.5-6.1, Top-10 ~20-29%  ❌ 嚴重 collapse
```

**訓練過程記錄**（實驗 1 為例）：

| Step | Entropy | Top-10 Mass | Strict Acc | Unique Tokens |
|------|---------|-------------|------------|---------------|
| 0 (初始) | **6.26** | 11.8% | 3.52% | 1583 |
| 200 | 5.50 | 26.7% | 0.65% | 1169 |
| 400 | 5.52 | 30.8% | 0.45% | 1285 |
| 600 | 5.53 | 29.8% | 0.52% | 1305 |
| 800 | 5.62 | 28.2% | 0.49% | 1318 |
| 1000 | **5.63** | **29.0%** | **0.60%** | 1437 |

**關鍵觀察**：
- ✅ 初始狀態健康（entropy 6.26 > baseline 6.07）
- ❌ 訓練 200 steps 後急速 collapse（entropy 6.26 → 5.50）
- ❌ 後續訓練無改善（entropy 在 5.5-5.6 之間振盪）

**從圖 2-3 可清楚看到**：
1. **Critical Period**：Collapse 主要發生在 step 0-200（訓練初期）
2. **持續惡化**：Top-10 mass 從初始 11.8% 飆升至 29%+
3. **Loss 收斂但 metrics 惡化**：Training loss 正常下降，但 collapse metrics 持續惡化
4. **兩實驗曲線幾乎重疊**：證明問題不在採樣策略

### 2.5 失敗原因深度分析

#### (1) 兩實驗結果驚人相似

| 指標 | 實驗 1 | 實驗 2 | 差異 |
|------|--------|--------|------|
| Entropy | 5.63 | 5.56 | 0.07 (1.2%) |
| Top-10 Mass | 29.0% | 28.3% | 0.7% |
| Strict Acc | 0.60% | 0.52% | 0.08% |

**結論**：兩種**完全不同**的採樣策略產生**幾乎相同**的失敗結果 → 問題不在採樣方法

#### (2) 採樣調整無效的根本原因

**實驗 1 失敗分析**：
- Proponents 不是病因（刪除反而惡化）
- Proponents 包含必要的困難樣本
- 軟性調整無法改變訓練目標和 loss landscape

**實驗 2 失敗分析**：
- 數據集已接近平衡（box 51.9% vs papercup 48.1%）
- papercup 過度代表是結果而非原因
- 強制平衡降低樣本多樣性

#### (3) 真正的問題：訓練目標與架構

**Loss Function 問題**：
- ❌ 缺乏 entropy regularization
- ❌ 缺乏 diversity constraint
- ⚠️ Intermediate supervision 可能過強

**VQ Codebook 問題**：
- ❌ Codebook 被凍結（未更新）
- ❌ 無 codebook refresh 機制
- ❌ 無 EMA 更新

**優化器問題**：
- ⚠️ 學習率過大（1e-4）
- ⚠️ Weight decay 不足（0.01）
- ❌ 無 gradient clipping on VQ

**架構問題**：
- ⚠️ LoRA rank 過大（256）
- ❌ 無 stochastic depth 正則化
- ❌ Encoder 缺乏 dropout

## 第三部分：Phase 2 建議方案

### 3.1 高優先級方案（立即測試）

#### 方案 A：Entropy Regularization

**方法**：在 loss 中加入 entropy 正則化項

```python
entropy = -torch.sum(token_probs * torch.log(token_probs + 1e-8), dim=-1).mean()
entropy_loss = -lambda_entropy * entropy  # Maximize entropy
total_loss = intermediate_loss + entropy_loss
```

**參數**：λ_entropy ∈ {0.01, 0.05, 0.1}
**預期**：明確懲罰低 entropy 分佈，鼓勵使用多樣 tokens
**實驗時長**：1000 steps (2-3 小時)

#### 方案 B：Codebook Refresh

**方法**：定期重置未使用的 codebook entries

```python
if step % refresh_interval == 0:
    usage_count = count_codebook_usage(codes, codebook_size=4096)
    unused_mask = usage_count < threshold
    codebook[unused_mask] = torch.randn_like(codebook[unused_mask])
```

**參數**：
- refresh_interval ∈ {50, 100, 200} steps
- threshold ∈ {5, 10, 20} minimum usage count

**預期**：防止 codebook collapse，維持 codebook diversity
**實驗時長**：1000 steps (2-3 小時)

### 3.2 中優先級方案（如果 A/B 失敗）

#### 方案 C：降低學習率

```python
lr = 5e-5          # 降低自 1e-4
warmup_steps = 100 # 更長 warmup
min_lr = 1e-7      # 更低 minimum
```

**預期**：減緩訓練速度，避免過早收斂
**實驗時長**：2000 steps (4-6 小時)

#### 方案 D：減小 LoRA Rank

```python
lora_rank = 128   # 降低自 256
lora_alpha = 256  # 保持 alpha/rank = 2
```

**預期**：限制模型容量，強制學習更 diverse 的編碼策略
**實驗時長**：1000 steps (2-3 小時)

### 3.3 成功判準

與 baseline 比較，需同時滿足：
```python
success = (
    entropy > 6.07 AND
    top_10_mass < 19.7% AND
    strict_acc >= 0.82%  # 90% of baseline
)
```

### 3.4 執行計劃

**短期驗證（1-2 天）**：
1. 測試方案 A（Entropy Reg）：λ ∈ {0.01, 0.05}
2. 測試方案 B（Codebook Refresh）：2-3 組參數

**中期驗證（3-5 天）**：
- 若成功 → Full training (300 epochs)
- 若失敗 → 測試方案 C & D

**長期方向（如果所有方案失敗）**：
1. 重新訓練 VQ Codebook
2. Multi-task Learning
3. Architecture Search

## 第四部分：關鍵洞察與貢獻

### 4.1 最重要發現

**Token collapse 是訓練動態問題，而非數據分佈問題**

證據：
1. ✅ 未訓練的 LoRA 優於訓練 300 epochs 的模型
2. ✅ 兩種完全不同的採樣策略產生相同失敗結果
3. ✅ 所有實驗在 early training (step 200) 就出現 collapse

**結論**：需要架構或 loss 層面的根本改變，採樣調整無效

### 4.2 方法學貢獻

**TracIn 診斷流程**：
1. ✅ 5-checkpoint 分析提高穩健性
2. ✅ L_train + L_anchor 雙版本交叉驗證
3. ✅ 音質評估 + bottom-PESQ 子集交叉
4. ✅ Counterfactual 驗證（避免誤判）

**實驗設計**：
1. ✅ 配置完全一致（僅 sampler 不同）
2. ✅ 詳細訓練動態記錄（每 200 steps）
3. ✅ 多維度失敗原因剖析（5 大類別）
4. ✅ 可執行的 Phase 2 方案（含參數範圍）

### 4.3 對 WavTokenizer 研究的啟示

**問題定位**：
- Token collapse 不是 WavTokenizer 架構問題
- 問題在於 LoRA fine-tuning 的訓練策略

**修正方向**：
- 短期：Entropy regularization + Codebook refresh
- 中期：優化器和 LoRA 配置調整
- 長期：考慮聯合訓練或多任務學習
